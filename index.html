Les repositories

Un respository (ou répertoire) est grossièrement un dossier de code en ligne. L'interêt est qu'une équipe de développeur puissent avoir accès au code source d'un projet et participer à l'élaboration de ce projet en y injectant leurs modifications, chacun travaillant depuis leur propre ordinateur sur une version locale du repository.

Créer un nouveau repository

Pour partir de rien, nous devons dans une premier temps créer un repository grâce à un outils de versionning en ligne, par exemple GitHub. Une fois ce dernier créé, nous pouvons créer une dossier dans les fichiers de notre ordinateur et définir ce dossier comme l'une des sources du repository.

Initialiser le repository

Une fois notre respository et dossier local créé, nous devons initialiser, c'est à dire définir notre dossier comme l'une des sources de ce repository.

git init

Créer une premier fichier et commit

Ce fichier - par exemple un fichier texte vierge - va permettre de réaliser une première participation au reposiroty. Nous pouvons ensuite l'ajouter dans un commit, qui sera donc le premier du repository

git add monFichier.txt

git commit -m "Premier commit"

Plus de détails sur les commits dans la section Envoyer du code sur un repository

Créer la branche principale

Lorsqu'on envoi du code sur un repository, on l'envoi sur une branche. Cette étape va permettre de créer la branche principale du repository, qui va généralement être nommé main ou master

git branch -M main

Dans ce cas, -M signifie --move --force. C'est utilisé afin de nous assurer que la branche principale est nommé comme nous le voulons, ici main

Plus de détails sur les branches dans la section Utiliser des branches

Ajouter le lien du repository en ligne

Maintenant que tout est prêt du coté de notre ordinateur, nous pouvons lier notre dossier au repository local. Pour cela, nous utilisons la commande suivante :

git remote add origin [lienDeNotreRepo]

remote spécifie que notre repository est accessible sur un serveur en ligne. add origin permet d'ajouter la source de ce repository, ici sous forme de lien. Le lien de notre repository est généralement accessible sur notre outils de versionning.

Ajouter le lien du repository en ligne

Dernière étape : Envoyer notre code sur le repository. Pour cela, on utilise la commande suivante :

git push -u origin main

Dans ce cas, -u est le diminutif pour --set-upstream, qui sert à ajouter un tracking sur toutes les branches crées jusqu'à maintenant, ici la branche main. origin main permet de trouver une référence à une possible branche main dans le repository et de la mettre à jour, ou la créer le cas échéant

Plus de détails sur les push dans la section Envoyer du code sur un reposiroty

Résumé

echo "# Mon Repo" >> README.md

git init

git add README.md

git commit -m "Premier commit"

git branch -M main

git remote add origin [lienDeMonRepo]

git push -u origin main

La première commande, echo "# Mon Repo" >> README.md, permet d'écrire dans la phrase entre guillemet dans un fichier, ici README.md. Ce fichier est créé s'il n'existe pas, ou modifié s'il existe.

Cloner un repository existant

Il est également possible de partir d'un repository existant. Pour travailler sur un tel repository, il faut d'abord le cloner afin de l'avoir à disposition localement, sur notre ordinateur. Pour cela, on utilise la commande suivante :

git clone [lienDuRepo]

Envoyer du code sur un repository

En tant que développeur, si je veux modifier le code d'un repository depuis mon ordinateur, j'ai besoin de connaître quelques commandes afin de partager mon travail.

Ajouter un changement

Afin d'envoyer les modifications apportées au code, il faut ajouter ces changements. Cette action est comparable à une sauvegarde dans un fichier Word par exemple.

git add .

Le . permet d'ajouter automatiquement tous fichiers contenant au moins une modification. On peut aussi ajouter un fichier en particulier comme suit :

git add monFichier.txt

Préparer un commit

Un commit est similaire à un paquet que l'on prépare afin de l'envoyer dans le repository. Avant d'envoyer du code dans le repository, il faut préparer un commit. La commande suivante va préparer un commit avec tout ce qui a été ajouté plus tôt grâce à la commande précédente.

git commit -m "message"

-m permet d'ajouter une message descriptif au commit.

Push les changements

Maintenant que les changements ont été ajouté et préparer en commit, on peut envoyer tous les commit sur le repository grâce à la commande suivante :

git push

NB : On peut push plusieurs commit d'un coup grâce à cette commande

Résumé

git add .

git commit -m "message"

git push

Récupérer du code depuis un repository

En tant que développeur, avant de me mettre à travailler localement sur mon ordinateur, je veux m'assurer d'avoir à ma disposition la dernière version en date du code disponible sur le repository.

Pull le code du repository

Pour récupérer le code disponible sur un repository, on utilise la comande suivante :

git pull

Utiliser des branches

Une branche est comparable à une copie du code sur laquelle nous pouvons travailler. Il est courant d'utiliser les branches afin de pouvoir effectuer des modifications sans impacter le code principal (la branche main ou master en général. Une fois notre version du code vérifié, nous pouvons ensuite merge notre branche sur la branche principale, afin que tous ai accès à la version à jour.

Lister les branches

Pour afficher toutes les branches du repository, on utilise la commande suivante :

git branch

Une emphase sera faite sur la branche sur laquelle on se trouve au moment de l'execution de la commande

Créer une branche

Pour ajouter une branche au repository, on utilise la commande suivate :

git branch [maBranche]

NB : Créer une branche de cette manière ne nous fera pas changer de branche, nous serons toujours sur la branche sur laquelle nous avons éxécuté la commande.

Changer de branche

Une fois notre nouvelle branche créée ou la branche sur laquelle nous voulons travailler identifiée, la commande suivante permet de se rendre dessus :

git checkout [maBranche]

Astuce : Il est possible de créer une branche et de se rendre directement dessus. Pour cela, nous disposons de la commande suivante :

git switch [maBranche]

Renommer une branche

Pour renommer une branche, on utiliser la commande suivante :

git branch -m [ancienNom] [nouveauNom]

Dans ce cas, -m est le diminutif pour --move

Supprimer une branche

Pour supprimer une branche, on utiliser la commande suivante :

git branch -d [maBranche]

Dans ce cas, -d est le diminutif pour --delete

Joindre notre branche à une autre

Une fois notre travail terminé et verifié sur notre branche, nous pouvons joindre notre version du code - sur notre branche - à la version sur la branche principale afin de la mettre à jour pour tous. Pour se faire, nous devons d'abord nous rendre sur la branche de destination avant de merge

git checkout main

git merge [maBranche]

Parfois, il faudra résoudre certains conflits, appellés merge conflicts. Cela arrive lorsque le serveur ne parvient pas à déterminer quelles lignes de codes sont à remplacer ou garder.

Example d'utilisation

Voici un exemple complet d'utilisation des commandes git, de la création d'un repository au merge d'une branche.

Exemple

Je commence par créer mon repository en ligne et executer la suite de commande pour préparer mon environnement local, et lier mon dossier avec mon repository en ligne.

echo "# Mon Repo" >> README.md

git init

git add README.md

git commit -m "Premier commit"

git branch -M main

git remote add origin [lienDeMonRepo]

git push -u origin main

Ensuite, je crée une branche afin de coder une fonctionnalité de manière sécuritaire, sans impacter ma branche principale.

git branch first-feature

git checkout first-feature

Une fois ma fonctionnalité codée et mon code vérifié, je peux envoyer mon code sur mon repository - et toujours sur ma branche first-feature - avant de merge ma branche sur la branche principale

git add .

git commit -m "First Feature OK"

git push

git checkout main

git pull

git merge first-feature

NB : Ici, la commande git pull utilisée après être retourné sur la branch main est une sécurité, au cas oú un autre développeur aurait merge sa propre branche entre temps.


Je peux maintenant aller dormir tranquillement, mon code est sur la branche principale, disponible pour les autres développeurs qui veulent participer au projet :)
